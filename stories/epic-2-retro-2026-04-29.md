# Epic 2 Retrospective: Become an Eligible Paid User

Date: 2026-04-29

## Scope Reviewed

Epic 2 goal: let an eligible user establish identity, pass safety screening, save a recovery profile, review a personalized summary, and start one-time checkout without leaking downstream program, AI, or analytics scope into the conversion path.

Stories reviewed:

- `2-1-magic-link-identity-session`: done
- `2-2-eligibility-safety-gate`: done
- `2-3-recovery-profile-multi-step-form`: done
- `2-4-personalized-summary-one-time-checkout`: done

No previous retrospective document was found for Epic 1, so this is the first recorded retrospective baseline.

## Project Lead Input

- Success experience to preserve: strictly advancing by story boundaries.
- Risk to fix before Epic 3: the data model and content structure must be frozen before implementation.

## Outcome Summary

Epic 2 completed the paid-user conversion foundation:

- Magic Link identity and dev-only login are available.
- `/onboarding` is protected and has a fixed three-step flow.
- Eligibility, safety stop states, and clinician-attention messaging are implemented.
- `RecoveryProfile` persistence is in place and tied to `session.user.id`.
- Personalized Summary, `$14.99 one-time` pricing, checkout CTA, Stripe Checkout session creation, and local dev-mock checkout are implemented.
- Success and cancelled checkout pages exist with scoped copy that does not claim program unlock.

The epic deliberately did not implement webhook unlock, `Purchase`, `Program`, `ProgramDay`, AI, RAG, analytics, i18n, native app work, billing portal, refund handling, or program content.

## What Went Well

### Story Boundaries Stayed Strong

The strongest success pattern was that each story owned a narrow slice and did not pull future work forward:

- Story 2.1 stopped at identity and minimal session fields.
- Story 2.2 stopped at deterministic eligibility and safety outcomes.
- Story 2.3 stopped at profile capture and persistence.
- Story 2.4 stopped at checkout session creation and honest post-checkout pages.

This kept the codebase understandable and prevented premature coupling between auth, billing, program generation, and AI.

### Safety and Compliance Were Kept Visible

The implementation consistently avoided medical-clearance language. The eligibility gate, profile flow, summary, and checkout success copy all preserve the educational-support boundary.

### Tests Became the Source of Confidence

Playwright coverage grew with the conversion path:

- Unauthenticated redirects
- Dev login
- Eligibility outcomes
- Profile validation and persistence
- Summary rendering
- Dev-mock checkout redirect
- Success and cancelled checkout states

The focused auth/onboarding suite became the reliable regression surface for Epic 2.

## What Was Difficult

### Environment Stability Repeatedly Surfaced as Product Risk

The main recurring friction was not business logic. It was local and E2E environment stability:

- Next.js 16 dev server behavior required `next dev --webpack` for stable E2E.
- Stateful auth/profile flows were flaky when Desktop and Mobile projects ran concurrently.
- Prisma persistence required real Postgres alignment and Supabase pooler access.
- Hydration timing could trigger native form behavior before React handlers were ready.

These were all resolved, but they are important signals for Epic 3 because webhook, pgvector, program creation, and richer persistence will increase the same class of risk.

### Story 2.3 Exposed Database Reality Earlier Than Planned

`RecoveryProfile` made it clear that SQLite/local shortcuts are not a safe default for business flows. Epic 3 should continue treating Supabase Postgres as the canonical development and validation environment, especially once `Purchase`, `Program`, `ProgramDay`, and `KnowledgeChunk` enter the schema.

### Some UX Polishing Was Correctly Deferred

Two non-blocking items remain documented in `stories/deferred-work.md`:

- Recovery-window upper-boundary copy can show a day-15 phrasing while still following current spec notes.
- Returning from Summary to Recovery Profile remounts the form and loses entered local state.

Neither blocks Epic 3, but both should stay visible for later onboarding polish.

## Lessons Learned

1. Keep story scope hard, especially around billing and program provisioning.
2. Treat Postgres parity as a default for any persistence-heavy story.
3. Keep auth/onboarding E2E serial while the suite mutates shared user/session/profile state.
4. Add hydration guards to critical form submit paths before debugging test flakes as Playwright-only issues.
5. Use dev-mock integrations only where the downstream provider is not the behavior under test.
6. Do not start Epic 3 by writing code into a vague schema; freeze the model and content contracts first.
